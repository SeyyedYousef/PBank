# 🚀 نقشه راه و اسناد معماری سوپراپ PBank (ویرایش ۲۰۲۶)

این سند، مرجع (Source of Truth) قطعی برای توسعه‌ی کامل پروژه دستیابی به سوپراپ PBank است.
هم کاربر (مدیر پروژه) و هم هوش مصنوعی در هر مرحله باید به این ساختار وفادار باشند و پیشرفت کار را بر اساس این مراحل (Phases) جلو ببرند.

---

## 🎯 فاز ۰: تکمیل و پولیش وب‌اپلیکیشن موقت (وضعیت فعلی: ۱۰۰٪ پایان یافته)
ما در این فاز یک دمو گرافیکی کامل (MVP فرانت‌اند) با ری‌اکت و فریم‌موشن ساختیم.

- [x] طراحی UI/UX با Tailwind CSS و Framer Motion (Glassmorphism، دیزاین مدرن)
- [x] طراحی کامپوننت‌های پایه (دکمه‌ها، کارت‌ها، اینپوت‌های بانکی)
- [x] پیاده‌سازی صفحات اصلی (خانه، انتقال، پروفایل، خدمات، گیمفیکیشن)
- [x] سیستم استیت‌منیجمنت موقت با Zustand (کیف پول، کاربران، جوایز)
- [x] بیلد نهایی و رفع باگ‌های ظاهری (`npm run build` موفقیت‌آمیز بود)

---

## 🏗️ فاز ۱: پایه‌ریزی سرور و دیتابیس (Core Backend)
**تکنولوژی‌های هدف:** Node.js, NestJS, TypeScript, PostgreSQL, Prisma ORM, **Docker Compose**, **GraphQL**
**مکان پروژه:** پوشه `PBank-backend` (جدا از فرانت‌اند)

این فاز قلب سیستم مالی است. هیچ کد تقلبی (Mock) نداریم و روی دیتابیس واقعی کار می‌کنیم.

- [ ] **گام ۱.۱: راه‌اندازی شاکله‌ی سرور (با داکر)**
  - نصب و مقداردهی اولیه NestJS با TypeScript
  - تنظیم **Docker Compose** برای اجرای ایزوله دیتابیس PostgreSQL و Redis (بدون نیاز به نصب دستی روی سیستم)
  - اتصال Prisma ORM به دیتابیس و کانفیگ اولیه
- [ ] **گام ۱.۲: معماری احراز هویت (Auth & KYC)**
  - طراحی مدل دیتابیس کاربر (`User Schema`: شماره موبایل، پسورد هش شده، نقش کاربر)
  - پیاده‌سازی سیستم ثبت نام/ورود (Login/Register Auth)
  - تنظیم توکن‌های امنیتی `JWT` (Access Token & Refresh Token)
  - ساخت API های تأیید هویت کاربری (KYC)
- [ ] **گام ۱.۳: هسته‌ی کیف‌پول (Wallet Engine) - [بسیار حساس]**
  - طراحی مدل‌های `Wallet` و `Transaction` (حسابداری دوطرفه یا ساختار لجر Ledger)
  - نوشتن اِی‌پی‌آی دریافت موجودی (Get Balance)
  - نوشتن توابع انتقال وجه (Transfer) با رعایت دقیق `Database Transactions` (قفل کردن ردیف‌ها برای جلوگیری از تراکنش همزمان و کسر موجودی اشتباه)
- [ ] **گام ۱.۴: ایجاد لایه API با GraphQL**
  - جایگزینی API های سنتی متنی با `GraphQL` برای دریافت منعطف و ده‌ها برابر سریع‌تر داده‌ها در موبایل (حل مشکل Over-fetching).
- [ ] **گام ۱.۵: اتصال به فرانت‌اند (Integration)**
  - اتصال فرم ورود (Login Web) به API های واقعی سرور
  - اتصال داشبورد به کیف پول واقعی و نمایش گردش حساب‌های ذخیره شده در Postgres

## ⚙️ فاز ۲: سرویس‌ها و گیمفیکیشن (Backend Services)
**تکنولوژی‌های هدف:** NestJS Modules, **Redis** (برای سرعت مافوق‌صوت)

- [ ] **گام ۲.۱: ماژول خدمات جانبی**
  - طراحی API برای خرید شارژ موبایل و پرداخت قبوض (تعریف مدل ServiceCategory و ServiceRequest)
- [ ] **گام ۲.۲: موتور گیمفیکیشن (Gamification Engine)**
  - طراحی مدل‌های پایگاه داده برای `XP`, `Levels`, `Badges` و `Rewards`
  - نوشتن منطق اهدای جایزه هنگام هر تراکنش جدید کاربر
  - استفاده از **Redis** برای سیستم Leaderboard (نمایش زنده‌ی برترین کاربران در حافظه موقت برای سرعت بالا)

---

## � فاز ۲.۵: صف‌های پیام و معماری رویداد‌محور (Event-Driven)
**تکنولوژی‌های هدف:** **Apache Kafka** یا **RabbitMQ**

وقتی روزانه صدها هزار درخواست شارژ موبایل ارسال شود، سیستم نباید مسدود (Block) شود! در سطح ابر-اپلیکیشن بانکی، تمام این تراکنش‌ها در یک «صفِ غیرخُردشونده» ذخیره و به نوبت انجام می‌شوند.

- [ ] راه‌اندازی **Kafka** برای مدیریت حجم عظیم تراکنش‌ها در صف پردازش (سرویس‌های بانکی به‌صورت لیسِنِر/Listener منتظر پیام می‌مانند).
- [ ] شروع خُرد کردن کدهای مونولیت به «**Microservices**» مجزا (سرویس کیف پول جدا از سرویس ثبت‌نام کاربر روی کانتینرهای مختلف).

## �📊 فاز ۳: پنل مدیریت کلان (Admin Dashboard)
**تکنولوژی‌های هدف:** React (Next.js یا Vite), Tailwind, Recharts

برای اینکه شعبه مرکزی شما (PBank) بتواند کاربرانش را کنترل کند، به یک ادمین‌پنل نیاز دارد.

- [ ] **گام ۳.۱: داشبورد ادمین**
  - ساخت یک فرانت‌اند جدید `PBank-admin`
  - صفحه چارت‌های درآمدی و آماری
- [ ] **گام ۳.۲: کنترل کاربران**
  - صفحه تایید/رد مدارک احراز هویت کاربران (KYC Approval)
  - مسدودسازی (Ban) و مشاهده لاگ تراکنش‌های افراد
- [ ] **گام ۳.۳: مدیریت اپلیکیشن**
  - تولید اطلاعیه‌ها (Notifications) که به گوشی کاربران پوش (Push) شود
  - تنظیم بنرهای تشویقی داخل اپلیکیشن از راه دور

## 📱 فاز ۴: تولد نهایی اپلیکیشن موبایل (React Native / Expo)
**تکنولوژی‌های هدف:** React Native, Expo, NativeWind, React Navigation
**مکان پروژه:** پوشه جدید `PBank-mobile`

حالا که موتور ماشین (سرور) ساخته شد و به خوبی با وب کار می‌کرد، آن طراحی بی‌نظیر را تبدیل به اپلیکیشن بومی می‌کنیم.

- [ ] **گام ۴.۱: استقرار معماری موبایل**
  - ایجاد ستاپ Expo با TypeScript
  - پیاده‌سازی سیستم روتینگِ موبایل (`React Navigation` بجای React-Router)
  - تکرار استایل‌های Tailwind با `NativeWind`
- [ ] **گام ۴.۲: بازنویسی لایه‌ی ظاهری به Native**
  - تبدیل تگ‌های `<div>/<span>` به `<View>/<Text>` در صفحات خانه، انتقال، و پروفایل
  - بازنویسی انیمیشن‌های Framer Motion به `react-native-reanimated` (برای پرفورمنس ۶۰ فریم بر ثانیه روی موبایل)
- [ ] **گام ۴.۳: دسترسی‌های سخت‌افزاری بومی (Native Features)**
  - استفاده از سنسورهای بیومتریک گوشی (اثرانگشت / FaceID) برای اجازه انتقال وجه (`expo-local-authentication`)
  - اسکنر بارکد بومی با دوربین موبایل برای بخش "پرداخت با QR"
  - خواندن کانتکت‌های گوشی (Contacts) کاربر برای ارسال پول به مخاطبین دفترچه تلفن
- [ ] **گام ۴.۴: اتصال به سرور تولید شده در فاز ۱**
  - جایگزینی Zustand Mock Data با رکوئست‌های واقعی Axios به سمتِ بک‌اندِ ما
  - بیلد گرفتن خروجی اندروید (APK/AAB) برای نصب روی گوشی‌ها

## 🚀 فاز ۵: نظارت و استقرار سازمانی (Monitoring & DevOps)
**تکنولوژی‌های هدف:** Docker, Kubernetes (K8s), **ELK Stack**, **Prometheus & Grafana**, Nginx, GitHub Actions

حالا برنامه باید برای اتصال صدهزار کاربر همزمان آماده شود. بدون داکر، اجرا کردن سوپراپ در سطح جهانی غیرممکن است.

- [ ] **Dockerizing:** کانتینر کردن کدها با `Docker` تا کدهای ما در سرور دقیقا مثل کامپیوتر شخصی ما اجرا شوند.
- [ ] **Container Orchestration:** تنظیم `Kubernetes` برای زمانی که اپلیکیشن وایرال شد، به طور خودکار سرورها را زیاد کند (Auto-scaling).
- [ ] مستقر کردن سیستم مانیتورینگ **Prometheus & Grafana** (برای دیدن داشبورد مصرف CPU، حافظه و تعداد تراکنش‌ها در ثانیه به صورت گراف‌های جذاب).
- [ ] ساخت سیستم لاگ‌گیری مرکزی با **ELK Stack (Elasticsearch, Logstash, Kibana)** (اگر هر لحظه یک خطای کوچک پیش بیاید، ادمین بتواند لاگ کل سرورها را در یک چشم به هم زدن جستجو و رفع کند).
- [ ] خرید/تنظیم سرور مجازی لینوکسی (VPS) و استقرار دیتابیس امن پشتِ فایروال.
- [ ] تنظیم `Nginx` بعنوان Reverse Proxy و گرفتن گواهینامه SSL ایمن (https).
- [ ] پایپ‌لاین CI/CD: تنظیم گیت‌هاب اکشن، تا به محض کامیت (Commit) برنامه‌نویس، سرور خودش را اتوماتیک با کدهای جدید بروز کند.
