# PBank Backend Architecture (معماری بک‌اند)
> **Pattern:** Domain-Driven Design (DDD) + Microservices
> **Goal:** Scalability & Independence

برای دستیابی به یک سیستم "بی‌نظیر" و قابل گسترش، ما از معماری **میکروسرویس** با رویکرد **DDD** استفاده می‌کنیم.

## 1. دامنه‌ها (Bounded Contexts)
ما سیستم را به "قلمرو"های مستقل تقسیم می‌کنیم. هر قلمرو دیتابیس و منطق خودش را دارد و هیچکس حق ندارد مستقیماً در کار دیگری دخالت کند.

### 1.1. سرویس هویت (Identity Service)
*   **مسئولیت:** شناخت کاربر (KYC)، ورود، امنیت، توکن‌ها.
*   **زبان:** Rust (برای امنیت حداکثری).
*   **Domain Entities:** `User`, `AuthToken`, `BiometricKey`.

### 1.2. سرویس کیف پول (Wallet Core)
*   **مسئولیت:** نگهداری موجودی، دفتر کل (Ledger)، انتقال داخلی.
*   **زبان:** Go (برای سرعت بالای تراکنش همزمان).
*   **Domain Entities:** `Account`, `Transaction`, `LedgerEntry`.

### 1.3. سرویس ابزارها (Utility Service)
*   **مسئولیت:** خرید شارژ، پرداخت قبض، اینترنت.
*   **زبان:** Node.js / TypeScript (برای ارتباط راحت با APIهای جانبی).
*   **Domain Entities:** `Bill`, `TopUp`, `BreshnaBill`.

### 1.4. سرویس نوتیفیکیشن (Owl Service)
*   **مسئولیت:** ارسال پیامک، پوش نوتیفیکیشن، ایمیل.
*   **معماری:** Event-Driven (گوش می‌دهد که چه اتفاقی افتاده تا خبر دهد).

---

## 2. ارتباطات (Communication)
*   **همگام (Sync):** برای چیزهای خیلی فوری از **gRPC** استفاده می‌کنیم (مثل چک کردن موجودی قبل از خرید).
*   **ناهمگام (Async):** برای اکثر کارها از **Kafka** یا **RabbitMQ** استفاده می‌کنیم.
    *   *مثال:* وقتی شارژ خریده شد، سرویس Wallet یک ایونت `WalletDebited` می‌فرستد. سرویس Utility آن را می‌گیرد و شارژ را می‌فرستد. سرویس Owl آن را می‌گیرد و پیامک می‌زند.

## 3. اصول DDD (قوانین)
*   **Aggregates:** هر تراکنش یک "Aggregate Root" است. تا وقتی کل تراکنش تایید نشود، هیچ تغییری در دیتابیس ثبت نمی‌شود (ACID).
*   **Value Objects:** پول (Money) فقط یک عدد نیست; یک آبجکت است که هم "مقدار" دارد هم "واحد ارز". (نمیشود دلار را با افغانی جمع کرد).
